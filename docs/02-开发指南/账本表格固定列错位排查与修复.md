# 账本表格固定列错位排查与修复

## 背景

账本模块的流水明细、日表格、重要记录都属于横向宽表。用户在浏览这类表格时通常有两个强需求：

1. 横向滚动时仍能知道当前行属于哪一天；
2. 横向滚动到最右侧后仍能直接操作或查看关键备注。

因此这几张表需要同时支持左侧固定列和右侧固定列。之前为了规避错位问题，一度取消了右侧固定列，但这会牺牲操作便利性，不是最终解法。

## 问题现象

问题主要出现在 Ant Design Vue 表格开启横向滚动和固定列后：

1. 横向滚动时，固定列的表头和正文单元格出现像素级偏移；
2. 右侧固定列尤其明显，滚动到最右侧后，表头左边框与正文左边框能看到几像素分裂；
3. 长备注、长内容在横向滚动时可能透到固定列下面，造成固定列遮挡不严的视觉问题；
4. 如果左侧固定列过多，例如同时固定日期、类型、分类，实际可滚动区域会被挤压，只剩两三列可看。

这类问题不一定在慢速、小幅拖动时明显，但快速拖动、滚动到底部或数据列较多时会非常影响表格规整度。

## 曾经的错误方向

### 取消右侧固定列

取消右侧固定列可以绕开右侧表头/正文错位，但会让用户横向滚动后失去操作入口：

- 流水明细不能直接编辑、删除；
- 重要记录不能直接查看、编辑、删除；
- 日表格看大量分类金额时，右侧备注不再稳定可见。

这只是降低暴露概率，不是解决问题。

### 左侧固定过多列

固定日期、类型、分类、标题等多列会让用户定位信息更充分，但代价是横向可用空间过少。账本表格更适合把固定列控制在最小范围：

- 流水明细：勾选列和日期足够定位行；
- 日表格：日期足够定位行；
- 重要记录：日期足够定位时间线索。

类型、分类、标题等业务信息应释放到普通滚动区域。

### 只靠 CSS 强盖背景

给固定列设置 `background`、`z-index`、`overflow: hidden` 可以解决一部分透字问题，但不能解决表头和正文边界不一致。错位的核心不是背景，而是表头和正文两套 DOM 的几何位置不一致。

## 根因分析

Ant Design Vue 的表格在固定表头、横向滚动和纵向滚动同时存在时，会生成多层结构：

- `.ant-table-header`：表头滚动区域；
- `.ant-table-body`：正文滚动区域；
- `.ant-table-cell-fix-left`：左固定单元格；
- `.ant-table-cell-fix-right`：右固定单元格；
- `.ant-table-cell-scrollbar`：表头为了补偿正文纵向滚动条而额外生成的占位列。

右侧固定列错位的关键点在于：

1. 正文区域真实滚动条宽度来自 `.ant-table-body.offsetWidth - .ant-table-body.clientWidth`；
2. 表头区域的 scrollbar 占位列宽度来自组件库内部计算；
3. 在当前项目环境下，这两个值可能不完全一致；
4. 右侧固定表头的 `right` 值使用了表头占位宽度，右侧固定正文的 `right` 值使用正文滚动区域；
5. 两者差值会表现为右侧固定列表头和正文边界错开。

一次浏览器实测中，关键数据类似：

```json
{
  "bodyClientWidth": 1189,
  "bodyOffsetWidth": 1199,
  "bodyScrollbarWidth": 10,
  "headerScrollbarCellWidth": 15,
  "rightHeaderBefore": "right: 15px",
  "rightBody": "right: 0px",
  "visualDiff": 4.6
}
```

这说明表头额外预留了约 15px，但正文真实滚动条只有约 10px，右固定表头就会比正文偏出几像素。

左侧固定列问题相对简单，因为左侧没有 scrollbar gutter 补偿。只要列宽和 `left` 值一致，通常可以稳定对齐。

## 最终解决方案

### 1. 固定列策略重新收敛

账本三张表的固定列策略调整为：

| 页面 | 左侧固定 | 右侧固定 | 说明 |
| --- | --- | --- | --- |
| 流水明细 | 勾选列、日期 | 操作 | 保留行定位和编辑/删除入口 |
| 日表格 | 日期 | 当日备注 | 横向查看分类金额时仍能看到备注 |
| 重要记录 | 日期 | 操作 | 保留时间定位和查看/编辑/删除入口 |

这样既减少了左侧固定列占用，又恢复了右侧操作便利性。

### 2. 固定列背景和裁剪继续保留

`BlogTable.vue` 中继续保留固定列的背景、层级和裁剪：

```css
.blog-table__body :deep(.ant-table-tbody > tr > td.ant-table-cell-fix-left),
.blog-table__body :deep(.ant-table-tbody > tr > td.ant-table-cell-fix-right) {
  background: var(--console-surface, #fff);
  border-color: var(--console-border, #f0f0f0);
  z-index: 4;
  isolation: isolate;
  clip-path: inset(0);
}
```

这部分解决的是固定列遮挡不严、滚动内容透出来的问题。

### 3. 同步表头 scrollbar gutter

组件先读取正文真实滚动条宽度：

```js
const scrollbarWidth = Math.max(body.offsetWidth - body.clientWidth, 0)
```

然后把表头 scrollbar 占位单元格和 `colgroup` 的最后一列都同步为这个宽度。这样可以减少组件库默认计算带来的差异。

### 4. 对右固定表头做几何校准

仅同步 gutter 还不够，因为 Ant Design Vue 可能在后续布局中再次写回内联 `right` 值。因此最终方案增加了二次校准：

1. 找到真实数据行，跳过 `.ant-table-measure-row`；
2. 找到所有右固定表头单元格；
3. 用同列正文单元格的实际坐标作为基准；
4. 如果表头和正文左边界差值超过 `0.5px`，就把差值补回到表头 `right`；
5. 使用 `style.setProperty('right', value, 'important')`，避免被组件库后续内联样式覆盖。

核心逻辑在 `frontend/src/components/BlogTable.vue`：

```js
const headerRect = cell.getBoundingClientRect()
const bodyRect = bodyCell.getBoundingClientRect()
const visualDelta = headerRect.left - bodyRect.left

if (Math.abs(visualDelta) > 0.5) {
  cell.style.setProperty('right', `${Math.max(0, expectedRight + visualDelta)}px`, 'important')
}
```

这个方案不再只猜滚动条宽度，而是直接以页面实际渲染结果为准。

### 5. 延迟到组件库布局之后再同步

Ant Design Vue 会在渲染后继续修正表格布局，所以同步逻辑需要延后执行。当前实现通过连续两个 `requestAnimationFrame`，让校准发生在组件库完成布局之后：

```js
fixedColumnFrameId = requestAnimationFrame(() => {
  fixedColumnFrameId = requestAnimationFrame(() => {
    fixedColumnFrameId = 0
    syncRightFixedHeaderOffsets(root, scrollbarWidth)
  })
})
```

如果只在第一次布局时修正，很容易被组件库后续写回的 `right: 15px` 覆盖。

## 验证方式

修复不能只靠肉眼看，需要在浏览器中读取 DOM 几何坐标。验证时分别进入：

```text
/console/ledger/entries
/console/ledger/daily
/console/ledger/moments
```

将 `.ant-table-body.scrollLeft` 设置到最大值，然后比较表头和正文同列的 `getBoundingClientRect()`。

本次验证结果：

| 页面 | 左固定列 | 右固定列 | 验证结果 |
| --- | --- | --- | --- |
| 流水明细 | 日期 | 操作 | 表头/正文左右边界偏差 `0px` |
| 日表格 | 日期 | 当日备注 | 表头/正文左右边界偏差 `0px` |
| 重要记录 | 日期 | 操作 | 表头/正文左右边界偏差 `0px` |

还确认了：

- 流水金额排序按钮会真实请求 `sortField=amount&sortOrder=asc/desc`；
- 日表格第一页仍按最新日期优先展示；
- 长备注和长内容不会再横向透到固定列下面。

## 后续注意事项

1. 账本宽表不要轻易增加左固定列。除非定位信息不足，否则优先只固定日期或勾选列。
2. 右侧操作列可以固定，但必须给列设置明确 `width`，并保持 `scroll.x` 大于列宽总和。
3. 新增长文本列时，需要使用省略裁剪容器，避免内容跨列透出。
4. 如果后续升级 `ant-design-vue`，需要重新验证 `BlogTable` 的右固定表头校准逻辑。组件库内部 DOM 或 scrollbar 补偿策略变化后，这段兼容逻辑可能需要调整。
5. 表格排序如果使用服务端排序，应优先做显式交互或确认 `a-table @change` 能真实触发请求，不要保留只有视觉图标但无行为的 sorter。
6. 修表格视觉问题时，建议同时做两类验证：肉眼拖动滚动条检查观感，以及 DOM 坐标差值检查边界是否真的对齐。
