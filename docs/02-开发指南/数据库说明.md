## 1. 当前使用的数据库

当前项目使用的是：

- 数据库：MongoDB
- ODM：Mongoose
- 后端连接配置位置：`backend/src/config/database.js`
- 运行时连接串来源：`backend/.env` 中的 `MONGODB_URI`

当前试用服务器使用的数据库连接串是：

```text
mongodb://127.0.0.1:27017/personal_fullstack_blog
```

这表示：

- 数据库服务运行在当前服务器本机
- 数据库名是 `personal_fullstack_blog`

## 2. 数据库里现在为什么是空的

这次从零部署时，我们只做了这些事：

- 安装 MongoDB
- 初始化管理员账号
- 启动前后端服务

但我们还没有执行“文章导入”或“业务数据初始化”动作。

所以当前数据库里虽然已经有：

- 管理员用户

但还没有这些业务数据：

- 文章
- 分类
- 标签
- 评论
- 媒体库内容
- 通知公告
- 站点设置

因此你现在登录后台后看到“暂无符合条件的文章”，这是正常现象，不是部署失败。

## 3. 为什么 `.env` 有管理员信息，还要执行创建管理员脚本

`backend/.env` 中的这几项：

```text
ADMIN_USERNAME
ADMIN_EMAIL
ADMIN_PASSWORD
```

只是给管理员初始化脚本提供参数，本身不会自动把管理员写入数据库。

真正把管理员写入 MongoDB 的命令是：

```bash
npm run create:admin
```

这个脚本会：

- 如果管理员邮箱不存在：创建一个管理员账号
- 如果管理员邮箱已存在：更新成管理员，并重置为 `.env` 中的密码

所以：

- `.env` 是原料
- `create:admin` 是执行动作

## 5. 首次部署时，数据库通常需要做什么

首次部署一般只需要做这几件事：

1. 安装并启动 MongoDB
2. 在 `.env` 中写好 `MONGODB_URI`
3. 初始化管理员账号
4. 如果项目有历史内容，再执行数据导入脚本

当前项目已经完成了前 3 步，但第 4 步还没做，所以文章为空。

## 6. 后续更新时，数据库通常要不要动

大多数普通前端更新：

- 不需要动数据库

普通后端接口或页面更新：

- 也不一定要动数据库

只有下面几种情况，数据库才需要额外处理：

- 新增了模型字段
- 写了数据迁移脚本
- 需要导入历史文章
- 需要初始化或修复某类数据

所以你之前感觉“前端、后端、数据库应该独立”，这个理解是对的。正常更新时，它们确实不总是一起动。

可以简单理解为：

- 前端：更新页面和交互
- 后端：更新接口和服务逻辑
- 数据库：只在数据结构或数据内容变化时处理
