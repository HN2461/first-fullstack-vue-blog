# 从零部署实操记录

本文档用于记录服务器重装后，从零到一部署本项目的实际操作过程。

记录规则：

- 每次只执行一个步骤，完成后再追加下一步。
- 每一步都记录：要做什么、怎么做、执行后怎么检查、实际结果、问题与解决方式。
- 不提前堆很多步骤，避免复盘时看不清当时到底做了什么。

## 第 1 步：确认服务器基础信息与登录方式

### 目标

确认当前服务器的系统版本、CPU 架构、登录账号、公网 IP，以及是否能通过 SSH 正常连接。

这一步不安装任何软件，也不改项目代码，只是确认后续部署环境。

### 你现在要做什么

在你本机打开 Xshell，新建或打开服务器连接，使用服务器公网 IP 登录。

登录成功后，在服务器终端执行下面这组命令：

```bash
whoami
hostname
uname -a
cat /etc/os-release
pwd
```

### 执行后怎么检查

重点看三件事：

- `whoami` 是否是预期账号，通常是 `root`
- `cat /etc/os-release` 是否能看出系统发行版和版本
- `pwd` 是否在你登录后的目录，比如 `/root`

### 预期结果

命令可以正常执行，并能看到类似下面的信息：

```text
root
服务器主机名
Linux ...
NAME="Alibaba Cloud Linux" 或 Ubuntu / CentOS 等系统信息
/root
```

### 实际结果

已确认服务器可以通过 Xshell 正常登录，当前环境信息如下：

```text

主机名：iZ0jlh0p2oqvhulhuhmytjZ

内核：5.10.134-18.al8.x86_64
架构：x86_64

```

### 遇到的问题

本步骤未出现异常。

### 最终怎么解决

无需额外处理，确认服务器基础环境正常，可以继续后续部署。

## 第 4 步：安装 MongoDB 8.0 数据库服务

### 目标

## 第 14 步：创建 Nginx 站点配置文件

### 目标

为当前项目写一份独立的 Nginx 配置，让它完成三件事：

- 托管前端静态文件 `/www/personal-blog/frontend`
- 把 `/api` 请求转发到后端 `127.0.0.1:3001`
- 把 `/uploads` 请求转发到后端 `127.0.0.1:3001`

### 你现在要做什么

在服务器执行下面命令，创建项目专用配置文件：

```bash
cat > /etc/nginx/conf.d/personal-blog.conf <<'EOF'
server {
    listen 80;
    server_name _;

    root /www/personal-blog/frontend;
    index index.html;

    client_max_body_size 20m;

    location / {
        try_files $uri $uri/ /index.html;
    }

    location /api/ {
        proxy_pass http://127.0.0.1:3001/api/;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    location /uploads/ {
        proxy_pass http://127.0.0.1:3001/uploads/;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}
EOF
```

### 执行后怎么检查

创建完成后，执行：

```bash

```

重点看三处内容是否正确：

- `root /www/personal-blog/frontend;`
- `/api/` 反向代理到 `127.0.0.1:3001`

## 第 49 步：修复知识库左侧目录接口职责过重问题

### 目标

把知识库左侧目录从“复用完整文章列表接口”调整为“使用独立的轻量菜单接口”，避免目录加载一次性拉取几百篇文章正文，导致前端请求体过大、浏览器中断请求。

### 问题原因

原先左侧目录逻辑会同时请求：

```text
/api/public/home
/api/public/articles?pageSize=500
```

其中第二个接口原本返回完整文章对象，包含：

- `contentMarkdown`
- `resources`

在服务器恢复 413 篇文章后，这个请求体膨胀到约 10MB，浏览器网络层出现 `ERR_ABORTED`，左侧目录因此显示“正在同步目录...”或“目录加载失败”。

### 本次改动

后端：

- 为公开侧新增独立轻量接口：`/api/public/knowledge-menu`
- 仅返回目录构建所需的 `categories` 和轻量 `articles`
- 菜单文章数据默认不再包含 `contentMarkdown` 和 `resources`

前端：

- `ConsoleLayout` 不再复用 `/api/public/articles?pageSize=500`
- 改为直接请求新的 `/api/public/knowledge-menu`

### 验证结果

本地验证已通过：

- 后端测试：`public-reader.test.js`、`knowledge-menu.test.js` 全部通过
- 前端生产构建成功
- 编码检查通过

### 说明

这是一次“做规范点”的结构性修复，不只是把接口瘦身，而是把“文章列表接口”和“目录菜单接口”的职责正式拆开。
