# 部署说明

本项目已经拆分为两个独立应用：

- `frontend/`：前端 Vue + Vite
- `backend/`：后端 Express + MongoDB

## 部署阅读顺序

如果你是第一次从零部署，建议先看这两份文档：

1. [从零部署实操记录.md](./从零部署实操记录.md)
2. 本文档

前者按步骤记录“执行什么、怎么检查、结果是什么”；后者说明最终部署结构和各模块职责。

## 正式服务器首次登录注意事项

正式购买阿里云轻量应用服务器后，如果 Xshell 连接时无法勾选“Password / 密码”认证方式，先不要判断为密码错误。

常见原因是服务器 SSH 配置禁用了密码认证：

```bash
sudo grep -E "PasswordAuthentication|PubkeyAuthentication" /etc/ssh/sshd_config
```

如果看到：

```text
PasswordAuthentication no
```

说明密码登录被关闭。需要临时开启密码登录时，执行：

```bash
sudo sed -i 's/PasswordAuthentication no/PasswordAuthentication yes/' /etc/ssh/sshd_config
sudo systemctl restart sshd
```

然后重新连接 Xshell，选择 `Password` 方式登录。

安全建议：

```text
首次部署可以先开启密码登录，方便安装环境。
部署稳定后再考虑配置 SSH 密钥登录，并视情况关闭密码登录。
服务器密码不要写入仓库、文档或聊天记录。
如需本机长期复用部署账号，请放入 `scripts/deploy.local.ps1` 这类已被 `.gitignore` 忽略的私有文件，不要提交。
```

## 前端

```bash
cd frontend
npm install
npm run build
```

构建产物在 `frontend/dist`。

本地发布包推荐从项目根目录统一生成：

```powershell
pwsh -File scripts\package-release.ps1
```

生成产物：

```text
release/frontend-dist.zip
release/backend-release.zip
```

其中后端发布包采用白名单打包，默认不包含 `.env`、`node_modules`、`uploads`。

## 本机私有部署配置

如果希望以后部署时不再重复口述服务器账号密码，请在本机创建私有文件：

```text
scripts/deploy.local.ps1
```

仓库已提供示例文件：

```text
scripts/deploy.local.example.ps1
```

该私有文件已被 `.gitignore` 忽略，可以写入：

```powershell
$env:DEPLOY_HOST = '121.41.11.78'
$env:DEPLOY_USER = 'root'
$env:DEPLOY_PASSWORD = '<你的服务器密码>'
```

注意：

```text
deploy.local.ps1 只保留在当前电脑，不提交仓库。
仓库文档、脚本示例和聊天中不再记录真实密码。
```

## 一键增量部署

项目根目录新增：

```text
scripts/deploy-release.ps1
```

默认会顺序执行：

```text
1. frontend npm run build
2. backend npm run test
3. scripts/check-encoding.ps1
4. scripts/package-release.ps1
5. 通过 Paramiko 上传前后端发布包
6. 服务器备份 MongoDB、frontend、backend、uploads、backend/.env
7. 替换前后端、安装后端依赖、pm2 restart、pm2 save
8. 校验公网首页和 /api/health
```

执行方式：

```powershell
pwsh -File scripts\deploy-release.ps1
```

如果只想直接复用已有发布包重新部署，可跳过检查和打包：

```powershell
pwsh -File scripts\deploy-release.ps1 -SkipChecks -SkipPackage
```

## 后端发布包规则

统一打包脚本为：

```powershell
pwsh -File scripts\package-release.ps1
```

后端发布包默认包含：

```text
backend/package.json
backend/package-lock.json
backend/ecosystem.config.cjs
backend/src
backend/tests
backend/.env.example
backend/README.md
backend/vitest.config.js
```

后端发布包默认排除：

```text
backend/.env
backend/node_modules
backend/uploads
backend/logs
backend/legacy-notes
```

后续新增后端代码时，优先放入 `backend/src` 下对应子目录，例如 `routes`、`services`、`models`、`validators`、`scripts`。这些文件会自动进入发布包。

如果新增的是 `backend` 根目录下的运行必需目录，例如 `templates`、`public`、`views`、`migrations`、`prisma`，必须同步更新 `scripts/package-release.ps1`，把该目录加入发布白名单。

MongoDB 独立运行，连接串在 `backend/.env` 中配置。

## 访问路径

- 前端：Nginx 静态托管 `frontend/dist`
- API：Nginx 反向代理 `/api` 到 `127.0.0.1:3001`
- 上传文件：Nginx 反向代理 `/uploads` 到 `127.0.0.1:3001`

## 前端热修复发布

如果发布后只修前端问题，例如登录页 JS 报错、接口参数错误、样式问题，不需要重新发布后端。

本地执行：

```powershell
cd frontend
npm run build
cd ..
pwsh -File scripts\check-encoding.ps1
pwsh -File scripts\package-release.ps1
```

上传：

```text
release/frontend-dist.zip
-> /www/personal-blog/backups/frontend-dist.zip
```

服务器执行：

```bash
ls -l /www/personal-blog/backups/frontend-dist.zip
rm -rf /www/personal-blog/frontend/*
unzip -o /www/personal-blog/backups/frontend-dist.zip -d /www/personal-blog/frontend
ls -la /www/personal-blog/frontend
```

浏览器验证时使用 `Ctrl + F5` 强制刷新，避免继续加载旧 JS。

## 发布后日志复查

发布后查看 PM2 日志：

```bash
pm2 logs personal-blog-api --lines 120
```

该命令会持续监听日志，看完后按 `Ctrl + C` 退出，不会停止后端服务。

重点判断：

- `GET/POST ... 200`、`304` 通常是正常请求。
- `401` 如果发生在未登录访问 `/api/auth/me`，通常是正常的登录态探测。
- `404 /uploads/...` 多半表示历史上传文件缺失，不代表后端服务失败。
- `400` 需要看具体接口，常见原因是前端请求参数超过后端校验范围。
- `500`、`ValidationError`、`CastError`、`Unhandled`、数据库连接错误需要优先处理。
