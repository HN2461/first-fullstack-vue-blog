# 数据库发布与迁移流程

本文档用于说明本项目 MongoDB 数据库在发布时应该怎么判断、怎么备份、怎么迁移、怎么验证、怎么回滚。

核心原则：

```text
普通代码发布：只备份数据库，不改数据库。
结构兼容变更：发布代码后观察，不跑迁移。
结构不兼容变更：先备份，再跑明确的一次性迁移脚本。
整库覆盖：只在明确要用本地数据替换线上数据时使用。
```

## 一、数据库发布不是每次都要做

本项目使用：

```text
MongoDB + Mongoose
```

MongoDB 没有传统 SQL 的固定表结构。很多后端模型字段新增后，旧数据即使没有该字段，也可以继续工作。

所以每次发布前先判断：

```text
这次代码改动是否要求旧数据必须被批量修改？
```

如果答案是否定的，就只需要：

```text
发布前备份数据库
发布代码
验证接口
不跑迁移
```

## 二、什么时候不需要迁移数据库

以下情况通常不需要迁移：

- 只改前端页面、样式、交互。
- 只改接口实现，但兼容旧数据。
- Mongoose model 新增非必填字段。
- 新字段有 `default`，旧数据不补也能正常读取。
- 页面能处理字段为空。
- 新增索引但不要求立即重建数据。
- 新增接口、查询条件、展示字段，但旧数据没有该字段也不会报错。

示例：

```text
Article 新增 isPinned，默认 false。
User 新增 bio，默认空字符串。
Setting 新增一个配置 key，读取不到时使用默认值。
```

这种发布只需要备份，不需要改库。

## 三、什么时候必须迁移数据库

以下情况需要写一次性迁移脚本：

- 字段改名。
- 字段类型变化。
- 旧枚举值不再被新代码识别。
- 新增必填字段，旧数据没有该字段。
- 分类、文章、媒体之间的关联方式变化。
- 用户、角色、菜单权限结构变化。
- 要批量补齐旧数据默认值。
- 要批量修复历史脏数据。
- 要删除或合并重复数据。
- 要把本地某批业务数据同步到线上。

示例：

```text
Article.category 从字符串改成 ObjectId。
User.role 单角色字段改成 roles 数组。
Category.name 从 "01_AI编辑器流" 批量改成 "AI编辑器流"。
Role/Menu 权限菜单新增后，管理员角色必须批量绑定新菜单。
Media 文件路径从 backend/uploads 迁到 /www/personal-blog/uploads。
```

这类变更不能只发布代码，必须有明确迁移脚本和验证命令。

## 四、怎么判断这次数据库到底改没改

发布前先检查这些目录是否有变更：

```powershell
git diff -- backend/src/models backend/src/services backend/src/scripts backend/src/validators backend/src/constants
```

重点看：

```text
backend/src/models       数据模型是否变了
backend/src/services     是否有读写数据的新逻辑
backend/src/scripts      是否新增迁移/修复脚本
backend/src/validators   接口参数是否变严
backend/src/constants    枚举值是否变化
```

如果 `models`、`constants`、`validators` 里出现下面变化，要特别谨慎：

```text
required: true
enum
default
unique
index
字段类型
字段名
引用关系 ObjectId
```

如果不确定，按保守流程处理：

```text
先备份数据库
发布代码
不执行迁移
观察接口日志和核心页面
只有出现旧数据不兼容时，再写迁移脚本处理
```

## 五、数据库发布的三种方式

### 方式 A：只备份，不迁移

适合大多数发布。

服务器执行：

```bash
RELEASE_DIR=/www/personal-blog/backups/release-YYYYMMDD-before-update
mkdir -p "$RELEASE_DIR"
mongodump --uri="mongodb://127.0.0.1:27017/personal_fullstack_blog" --out="$RELEASE_DIR/mongodb-before"
```

检查：

```bash
find "$RELEASE_DIR/mongodb-before" -maxdepth 2 -type d
```

应看到：

```text
$RELEASE_DIR/mongodb-before
$RELEASE_DIR/mongodb-before/personal_fullstack_blog
```

然后发布前后端代码，验证接口即可。

### 方式 B：备份后跑迁移脚本

适合字段、权限、分类、媒体路径等需要批量修复的发布。

迁移脚本应放在：

```text
backend/src/scripts
```

脚本推荐支持两个模式：

```text
默认预览：只打印将要修改什么，不写数据库。
--apply：正式写入数据库。
```

推荐脚本命名：

```text
repairSomething.js
migrateSomething.js
fixSomething.js
```

服务器执行顺序：

```bash
cd /www/personal-blog/backend
node src/scripts/xxx.js
node src/scripts/xxx.js --apply
```

执行前必须确认已经有：

```text
$RELEASE_DIR/mongodb-before
```

### 方式 C：整库覆盖

只在明确要用本地数据库替换线上数据库时使用。

适用场景：

```text
试用服务器初始化。
线上没有新增数据。
本地数据库才是最新完整数据。
用户明确接受线上数据会被覆盖。
```

本地导出：

```powershell
$env:Path += ";E:\MongoDB\Tools\bin"
mongodump --uri="mongodb://127.0.0.1:27017/personal_fullstack_blog" --out="C:\Users\HN246\Desktop\全栈\个人全栈博客系统\mongodb-dump"
Compress-Archive -Path "C:\Users\HN246\Desktop\全栈\个人全栈博客系统\mongodb-dump\*" -DestinationPath "C:\Users\HN246\Desktop\全栈\个人全栈博客系统\mongodb-dump.zip" -Force
```

上传到服务器：

```text
/www/personal-blog/backups/mongodb-dump.zip
```

服务器恢复：

```bash
rm -rf /www/personal-blog/backups/mongodb-dump
unzip -o /www/personal-blog/backups/mongodb-dump.zip -d /www/personal-blog/backups/mongodb-dump
mongodump --uri="mongodb://127.0.0.1:27017/personal_fullstack_blog" --out="$RELEASE_DIR/mongodb-before"
mongorestore --uri="mongodb://127.0.0.1:27017" --drop /www/personal-blog/backups/mongodb-dump
pm2 restart personal-blog-api
```

注意：

```text
mongorestore --drop 会覆盖线上数据库。
发布后新增的文章、评论、用户、媒体记录都会丢失。
正式环境不要把整库覆盖作为常规发布方式。
```

## 六、迁移脚本应该怎么写

迁移脚本必须满足：

- 默认 dry-run，不写库。
- `--apply` 才真正写库。
- 打印修改前后的数量。
- 修改前创建 JSON 备份，或依赖发布前 `mongodump`。
- 只处理明确范围的数据。
- 不吞掉错误。
- 修改后重新统计关键集合数量。

脚本模板：

```js
import { connectDatabase, disconnectDatabase } from '../config/database.js'
import { Article } from '../models/Article.js'

const args = new Set(process.argv.slice(2))
const APPLY = args.has('--apply')

async function main() {
  await connectDatabase()

  const matched = await Article.find({
    // 写清楚筛选条件
  }).lean()

  console.log(`模式: ${APPLY ? '正式执行' : '预览'}`)
  console.log(`匹配数量: ${matched.length}`)

  if (APPLY) {
    // await Article.updateMany(...)
  }

  await disconnectDatabase()
}

main().catch((error) => {
  console.error('ERR:', error)
  process.exitCode = 1
})
```

## 七、迁移后怎么验证

基础数量检查：

```bash
mongosh mongodb://127.0.0.1:27017/personal_fullstack_blog --eval "print('articles=' + db.articles.countDocuments()); print('categories=' + db.categories.countDocuments()); print('tags=' + db.tags.countDocuments()); print('users=' + db.users.countDocuments()); print('roles=' + db.roles.countDocuments()); print('menus=' + db.menus.countDocuments());"
```

根据迁移内容补充专项检查。

例如检查角色和菜单：

```bash
mongosh mongodb://127.0.0.1:27017/personal_fullstack_blog --eval "db.roles.find({}, {name:1, code:1, menuIds:1}).pretty(); db.menus.countDocuments();"
```

例如检查文章分类：

```bash
mongosh mongodb://127.0.0.1:27017/personal_fullstack_blog --eval "db.categories.find({}, {name:1, parent:1, sortOrder:1, status:1}).limit(20).pretty();"
```

例如检查媒体文件路径：

```bash
mongosh mongodb://127.0.0.1:27017/personal_fullstack_blog --eval "db.media.find({}, {url:1, path:1, filename:1}).limit(20).pretty();"
```

页面验证：

```text
/console/articles
/console/manage/articles
/console/manage/categories
/console/manage/users
/console/manage/roles
/console/manage/media
```

日志验证：

```bash
pm2 logs personal-blog-api --lines 120
```

## 八、数据库回滚

数据库回滚只在确认数据被改坏时使用。

执行：

```bash
mongorestore --uri="mongodb://127.0.0.1:27017" --drop "$RELEASE_DIR/mongodb-before"
pm2 restart personal-blog-api
```

风险：

```text
会恢复到发布前数据库状态。
发布后新增的文章、评论、用户、媒体记录都会丢失。
```

所以回滚顺序建议：

```text
先回滚前端/后端代码
如果仍然是数据被改坏，再回滚数据库
```
