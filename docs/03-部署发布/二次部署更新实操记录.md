# 二次部署更新实操记录

本文档用于记录项目首次部署成功后，第二次进行功能更新、重新打包、重新上传、重新部署的实际过程。

记录规则：

- 每次只执行一个更新步骤，完成后再进入下一步。
- 每一步都记录：要做什么、怎么做、执行后怎么检查、实际结果、问题与解决方式。
- 重点记录“和首次部署有什么不同”，方便后面复盘增量更新流程。

## 第 1 步：本地重新打包前端

### 目标

把本次前端和后端接口调整后的前端代码重新构建，生成新的静态资源包。

这一步是二次部署的起点。前端这次主要是跟上后端新增的知识菜单接口。

### 你现在要做什么

在本机项目根目录执行：

```bash
cd frontend
npm run build
```

### 执行后怎么检查

构建完成后，执行：

```bash
dir dist
```

重点看：

- `dist` 是否重新生成
- `index.html` 和 `assets` 是否存在

### 预期结果

命令执行成功后：

- 前端新版本静态资源生成
- 后续可以重新压缩并上传到服务器

### 实际结果

前端构建成功，`dist` 目录已生成，检查结果如下：

```text

  assets/
  index.html
```

### 遇到的问题

本步骤未出现异常。

### 最终怎么解决

无需额外处理，新的前端静态资源已经生成，可以进入压缩上传步骤。

同时再次发现压缩包中包含：

```text
node_modules/
```

### 遇到的问题

当前压缩方式仍然把本地 `node_modules` 和 `.env` 一并带到了服务器。

这不影响继续部署，但不符合我们要固定下来的标准增量发布流程：

- `node_modules` 应在服务器本地重新安装
- `.env` 应在服务器独立维护，不依赖压缩包覆盖

### 最终怎么解决

继续沿用首次部署时确认过的规范处理方式：

- 删除服务器上的现成 `node_modules`
- 保留服务器现有 `.env`
- 在服务器重新执行 `npm install`
