## 一、基础概念

> 先理解"AI 到底是什么"，再谈"怎么用 AI"。

### 1.1 LLM（大语言模型）

**LLM = Large Language Model，大语言模型**

LLM 不是某个具体产品的名字，是**一类模型的统称**，就像"手机"是一个类别，不是某个品牌。

**本质：一个超级概率计算器**

- 你输入文字 → 内部几千亿个参数在算"下一个词，A 出现的概率是 0.3，B 是 0.15，C 是 0.08……" → 选概率最高的词依次输出
- 本质是"下一个词该说什么"的概率游戏
- 参数越多（几百亿个），上下文算得越准，看起来越像真人"在思考"

**重要认知：LLM 不是图书馆**

更准确地说，不是"图书馆里查资料"，而是**把文字压缩成数字，再去算概率**。它不知道"事实"，只懂"什么词该接什么词"。

你用过的都是 LLM：ChatGPT、Claude、通义千问、Kimi、MiniMax——只是公司和技术细节不同。

### 1.2 Agent（智能体）

LLM 只能回答问题，**Agent 能帮你做事**。

**LLM vs Agent**

| | LLM | Agent |
|--|-----|-------|
| 对你说"帮我整理文件" | 说"你可以用 Python 脚本……"（光说不做） | 自己规划步骤，读文件→判断日期→创建文件夹→移动文件→汇报完成（直接做掉） |
| 本质 | 输出文字 | LLM + 执行层（工具+记忆+规划） |

**Agent 多出的三样东西**

1. **工具（Tools）** — 能调用函数、查数据库、执行代码
2. **记忆（Memory）** — 记得之前聊过什么
3. **规划（Planning）** — 把大任务拆成小步骤，自己决定先做啥

**Agent 本质**

- LLM 告诉 Agent"这段代码该改成……"
- Agent 识别出这是代码修改指令
- Agent 调用工具（读文件、写文件、执行命令）
- **本质就是大量的 if-else 分支，分支足够多、执行足够快 → 看起来像真人**
- 没有真正的"理解"或"思考"，只是在走流程
- 工具越多，它能干的事越多

你用过的就是 Agent：Cursor、Copilot——能帮你直接写代码、改文件，不只是回答问题。

### 1.3 切片（Chunking）

**是什么**：把一整本书/文档切成一小段一小段。

**为什么切**：
- LLM 一次能"看"的字数有限（叫"上下文窗口"）
- 切成小段之后，才能精确找到用户问的那部分内容

**怎么做**：规定每 N 个字切一刀，或者按段落切。可以用代码直接实现，就是普通的文本处理逻辑。

### 1.4 向量化（Embedding）

**是什么**：每段话，用一个专门的 **Embedding 模型**（不是 LLM 本身）生成一串数字，这串数字叫**向量**或 **embedding**。

**Embedding 模型**：是**独立于 LLM 的专门模型**，职责只有一个：把文字变成一串数字。

| 公司 | Embedding 模型 |
|------|--------------|
| OpenAI | text-embedding-ada-002、text-embedding-3-small |
| MiniMax | 有自己的 embedding 模型 |
| 开源的 | BGE、all-MiniLM-L6-v2 |

Embedding 模型和 LLM 可以完全来自不同公司。

**向量的特点**：意思相近的内容，生成的数字向量也相近；意思完全不同的，数字差很远。

```
"今天天气很好，阳光明媚" → [0.12, -0.34, 0.78, 0.23, ...]
"外面太阳很大，适合出门" → [0.11, -0.33, 0.79, 0.22, ...]  ← 数字很接近
"苹果手机多少钱"         → [0.95, 0.02, -0.11, 0.07, ...]  ← 数字差很远
```

**切片 + 向量化什么时候做**：都是在"把文档放进系统"的时候**离线做完**的，不是用户提问时才做。

### 1.5 RAG（检索增强生成）

**RAG = Retrieval-Augmented Generation**

三个单词对应流程的三个步骤：

1. **Retrieval（检索）** — 从你的资料库里，找到跟问题相关的内容
2. **Augmented（增强）** — 把找到的片段，塞给 LLM，作为它的"参考资料"
3. **Generation（生成）** — LLM 读完参考资料，生成回答

**重要结论：RAG 不是一个独立的技术，是一套流程/工作流的名字。**

中间那些步骤——切片、向量化、相似度匹配——每一步都有自己独立的名字和技术，RAG 只是把它们串起来之后给这一整套流程起的名字。

就像"外卖"不是一个技术，而是"下单→商家接单→骑手取餐→配送→收货"这一整套流程的统称。

**完整 RAG 流程**

```
文档 → 切片 → 向量化 → 存向量数据库
                           ↓
用户提问 → 向量化 → 在数据库里找相似片段 → 片段+问题 → 发给 LLM → 回答
```

**为什么需要 RAG**：LLM 训练时看的是网上公开的资料，你公司内部文档它没读过。有 RAG 之后，系统先帮你找到相关内容，再让 LLM 读，它就能回答了。

### 1.6 模型参数（1B、7B、70B）

**B = Billion = 10亿**
- **1B** = 10亿个参数
- **7B** = 70亿个参数
- **70B** = 700亿个参数

**参数是什么**：LLM 在"算概率"的过程中，每一步做乘法、加法用的那些数字，就是参数。参数越多，模型越"聪明"。

**规模参考**

| 参数规模 | 能跑在哪 | 大概文件大小 |
|---------|---------|------------|
| 1B~3B | 普通电脑可以跑 | 几 GB |
| 7B~14B | 需要好一点电脑，或用 Ollama | 十几 GB |
| 70B+ | 需要服务器级别配置 | 100GB+ |

### 1.7 Ollama：本地运行大模型

**是什么**：让你**在自己电脑上跑大模型**的工具。

**云端 vs 本地**

| | 云端跑模型 | Ollama 本地跑 |
|--|-----------|--------------|
| 怎么用 | 去 OpenAI、MiniMax 服务器上跑 | 下载到自己的电脑，直接跑 |
| 数据 | 对话传到外面服务器 | 数据不出本机 |
| 速度 | 快（服务器算力强） | 取决于你电脑的显卡 |
| 费用 | 要付费 | 免费（但要有好显卡） |

**怎么用**

```
1. 下载安装 Ollama
2. 终端输入：ollama run qwen2.5（或其他模型名）
3. 开始对话
```

**常见可跑模型**：qwen2.5（通义千问）、llama3、deepseek-r1 等

## 二、开发基础

> 理解了概念之后，开始动手调 API。

### 2.1 cURL：开发者必备的命令行工具

**是什么**：cURL（Client URL）是一个利用 URL 语法在命令行下工作的**开源文件传输工具**，1997 年首次发行，被广泛应用于 Unix、Linux、Windows 和 macOS 等操作系统。

**核心特点**：
- **跨平台性**：几乎所有操作系统都内置或可安装
- **协议广泛**：支持 100+ 协议（HTTP/HTTPS、FTP、SMTP、SCP 等）
- **轻量简洁**：无需复杂配置，即开即用
- **脚本友好**：完美适合自动化和 API 测试

**主要应用场景**：
- **API 测试与调试**：快速验证接口是否正常工作
- **文件传输**：下载/上传文件到服务器
- **网络诊断**：查看请求响应头、分析通信过程
- **数据采集**：简单的网页内容抓取
- **脚本自动化**：集成到 Shell/Python 脚本中

**常用命令速查**：

```bash
# 1. 获取网页内容
curl https://www.example.com

# 2. 下载文件并保留原名
curl -O https://example.com/file.zip

# 3. 发送 POST 请求
curl -X POST -d "key=value" https://api.example.com

# 4. 显示响应头信息
curl -I https://www.example.com

# 5. 跟随重定向
curl -L https://example.com

# 6. 显示详细通信过程
curl -v https://www.example.com

# 7. 设置请求头
curl -H "Content-Type: application/json" https://api.example.com

# 8. 发送 Cookie
curl -b "name=value" https://www.example.com
```

### 2.2 OpenAI 协议：AI 服务的事实标准

**是什么**：当 AI 服务商说"**兼容 OpenAI 协议**"时，意思是他们的 API 接口设计得和 OpenAI 一样，开发者可以无缝迁移。

**OpenAI 协议的核心组成**：

**① API 端点结构**
- `/v1/chat/completions` — 聊天补全（对话式交互）
- `/v1/completions` — 文本补全
- `/v1/embeddings` — 文本向量化
- `/v1/images/generations` — 图像生成
- `/v1/audio/transcriptions` — 语音转文本

**② 认证方式**

```http
Authorization: Bearer YOUR_API_KEY
```

**③ 请求体结构（聊天补全示例）**

```json
{
  "model": "gpt-3.5-turbo",
  "messages": [
    {"role": "system", "content": "你是一个有帮助的助手"},
    {"role": "user", "content": "你好，请介绍一下自己"}
  ],
  "temperature": 0.7,
  "max_tokens": 1000,
  "stream": false
}
```

**④ 响应格式**

```json
{
  "id": "chatcmpl-123",
  "object": "chat.completion",
  "created": 1677652288,
  "model": "gpt-3.5-turbo",
  "choices": [{
    "index": 0,
    "message": {
      "role": "assistant",
      "content": "你好！我是AI助手..."
    },
    "finish_reason": "stop"
  }],
  "usage": {
    "prompt_tokens": 9,
    "completion_tokens": 12,
    "total_tokens": 21
  }
}
```

**为什么 OpenAI 协议成为标准？**
- **先发优势**：OpenAI 是最早提供商业化大模型 API 的公司
- **设计优秀**：RESTful 架构，简洁直观
- **生态完善**：官方 SDK + 社区工具 + 框架集成
- **开发者友好**：简单认证、直观对话模型、丰富参数

**"兼容 OpenAI 协议"的实际意义**：

```python
# 原本的 OpenAI 代码
import openai
openai.api_key = "sk-openai-key"
response = openai.ChatCompletion.create(
    model="gpt-3.5-turbo",
    messages=[{"role": "user", "content": "你好"}]
)

# 切换到兼容服务商，只需修改：
openai.api_base = "https://api.兼容服务商.com"  # 修改端点
openai.api_key = "sk-兼容服务商-key"           # 修改密钥
# 其他代码完全不变
```

### 2.3 Prompt：提示词的完整理解

**Prompt = 提示词 = 你发给 AI 的那段话**

**Prompt 的两种类型**：
- **System Prompt（系统提示词）**：`system` 角色里的内容，设定 AI 的人设和行为
- **User Prompt（用户提示词）**：`user` 角色里的内容，具体的问题或需求

**完整示例**：

```
System Prompt（设定人设）：
"你是一个专业的 Python 程序员，回答要简洁，代码要加注释"

User Prompt（具体问题）：
"帮我写一个快速排序"
```

### 2.4 消息角色体系

**核心三角色（最常用）**：

| 角色 | 作用 | 示例 |
|------|------|------|
| `system` | 设定 AI 人设、行为准则 | "你是一个友好的客服" |
| `user` | 用户的真实问题 | "怎么退款？" |
| `assistant` | AI 的回复内容 | "您可以这样操作退款..." |

**扩展角色（按需了解）**：
- **`tool` / `function`**：函数调用结果，用于 Agent 开发
- **`developer`**：OpenAI 新引入，开发者技术配置（o1 系列优先看这个）

### 2.5 Token：AI 计费的最小单位

**什么是 Token？**

Token 是 LLM 处理文字的**最小单位**，不是字也不是词，是模型自己切分的小块。

**中英文 Token 对照**：
- **英文**：1 个 token ≈ 0.75 个单词 ≈ 4 个字母
- **中文**：1 个 token ≈ 1.2 个汉字

```
"I love you" → ["I", "love", "you"] → 3 个 token
"今天天气真好" → ["今天", "天气", "真", "好"] → 3-4 个 token
```

**为什么 Token 重要？**
- **所有计费按 token 算**：输入输出分开计费
- **上下文窗口以 token 为单位**：超出就"忘记"前面内容
- **模型不同 token 切分不同**：同样文字可能 token 数不一样

### 2.6 上下文窗口：AI 的"工作记忆"

**直观理解**

把模型的"上下文窗口"想象成一张**草稿纸**：
- AI 在这张纸上**写**（它的回复）
- 也在这张纸上**读**（你的对话）
- 草稿纸**有大小限制**
- 写满了、读不下了，就要"翻页"——前面的内容**就忘了**

**K 的含义**

> K = 千 = 1000，1K = 1000 个 token，数字越大能处理的内容越多

**具体容量对照表（实物类比）**：

| Token 数 | 中文大概 | 英文大概 | 实物类比 |
|---------|---------|---------|---------|
| 100 | 80 个汉字 | 75 个单词 | 一段话（2-3 句话） |
| 500 | 400 个汉字 | 375 个单词 | 一封短信 |
| 1K | 800 个汉字 | 750 个单词 | 一篇短新闻 |
| 2K | 1600 个汉字 | 1500 个单词 | 一篇博客文章 |
| 4K | 3200 个汉字 | 3000 个单词 | 一篇深度文章 |
| 8K | 6400 个汉字 | 6000 个单词 | 一篇论文 |
| 16K | 12800 个汉字 | 12000 个单词 | 一本短篇小说 |
| 32K | 25600 个汉字 | 24000 个单词 | 一本中篇小说 |
| 100K | 80000 个汉字 | 75000 个单词 | 一本长篇小说 |
| 200K | 16 万汉字 | 15 万单词 | 三体全集 |
| 1M | 80 万汉字 | 75 万单词 | 一座小型图书馆 |

**主流模型上下文窗口对比**：

| 模型 | 窗口大小 | 适合场景 |
|------|---------|---------|
| GPT-3.5 | 4K / 16K | 日常聊天、简单任务 |
| GPT-4 | 8K / 32K / 128K | 复杂推理、长文档 |
| Claude 3 | 200K | 超长文档分析 |
| Gemini 1.5 | 1M / 2M | 处理整个代码库/多本书 |

**实际场景消耗估算**：
- **聊 10 分钟** = 不到 1K token = 半页纸
- **写一封邮件** = 约 500-600 token = 一张 A4 纸
- **分析一篇文章** = 约 10K token = 2-3 张 A4 纸
- **读一本书** = 约 150K token = 一本中篇小说
- **读整个项目代码** = 几百万 token = 1000 页 A4 纸

**简单心算方法**：
- **1K token** ≈ 1 页 A4 纸（中文约 800 字）
- **10K token** ≈ 一篇深度文章（中文约 8000 字）
- **100K token** ≈ 一本小说（中文约 8 万字）

看到"K"反应是"几页纸"，看到"128K"反应是"一本小说"，看到"1M"反应是"一套书"。

### 2.7 Temperature：控制输出的"创造力"

**是什么？**

控制输出随机性的参数，0-2 之间。

**直观理解**：

| Temperature 值 | 效果 | 适用场景 |
|-------------|------|---------|
| **0** | 每次回答几乎一样 | 写代码、翻译、数据提取 |
| **0.7** | 平衡创造性和准确性 | 日常对话、内容创作 |
| **1.5+** | 很有创意但可能胡说八道 | 写小说、头脑风暴 |

**相关参数**：
- `top_p`：从概率最高的 token 中按比例选
- `frequency_penalty`：降低重复用词
- `presence_penalty`：鼓励新话题

### 2.8 幻觉（Hallucination）：AI 的一本正经胡说八道

回顾 1.1 节的 LLM 本质——LLM 是"概率计算器"，不是"知识库"，它不知道"事实"，只懂"什么词该接什么词"。这就是幻觉的根源。

**典型表现**：
- 编造不存在的论文
- 引用错误的数据
- 编造历史事件

**应对方法**：
- **RAG**：让 AI 基于真实资料回答（详见 1.5 节）
- **要求标注来源**：让 AI 注明信息来源
- **承认不知道**：让 AI 对不确定的内容明确说明

## 三、工程化实战

> 从"能跑"到"能上线"，需要完整的工程能力。

### 3.1 三大 HTTP 请求方式

| 方式 | 适用场景 | 特点 |
|------|---------|------|
| **cURL** | 命令行测试、快速验证 | 无需编程，即开即用 |
| **Python requests** | 脚本自动化、简单集成 | 最直接、最灵活 |
| **官方 SDK** | 生产项目、复杂应用 | 封装完善、功能丰富 |

### 3.2 错误处理与重试机制

网络抖动、限流、服务故障是常态，**必须做好错误处理**。

**常见错误类型**：

| 错误码 | 含义 | 处理方式 |
|--------|------|---------|
| **429** | 请求太快被限流 | 等几秒后重试 |
| **500** | 服务器内部错误 | 服务挂了，自动重试 |
| **超时** | 网络不好 | 换个姿势重试 |
| **Token 超限** | 输入内容超过窗口 | 减少输入或换大窗口模型 |

**重试机制示例**：

```python
def call_api_with_retry(prompt, max_retries=3):
    for i in range(max_retries):
        try:
            return call_api(prompt)
        except RateLimitError:
            time.sleep(2 ** i)  # 指数退避
        except ServerError:
            time.sleep(5)
    return None
```

### 3.3 结构化输出（JSON Mode）

让 AI 返回可解析的数据，而不是自由文本。

**两种方式**：
- **Prompt 约束**：在提示词中要求"请用 JSON 格式输出"
- **JSON Mode**：OpenAI 原生功能 `response_format: {"type": "json_object"}`

**实战价值**：
- AI 返回用户信息 → 直接入库
- AI 返回分类结果 → 直接处理
- AI 返回 SQL → 直接执行

### 3.4 文本预处理与多格式处理

**常见处理任务**：
- **去除噪声**：HTML 标签、特殊字符、多余空白
- **格式化**：统一编码、换行符
- **分块策略**：按段落、按句子、按 token 数
- **元数据提取**：标题、作者、时间

**多文件格式处理工具**：

| 格式 | Python 库 |
|------|----------|
| PDF | pdfplumber、PyPDF2 |
| Word | python-docx |
| Excel | pandas、openpyxl |
| 图片 OCR | Tesseract、PaddleOCR |
| 音视频 | Whisper、ffmpeg |
| 网页 | BeautifulSoup、Playwright |

### 3.5 缓存策略：省钱省时间的关键

**缓存什么？**
- **完全相同的问题**：直接返回缓存
- **相似问题**：用 embedding 找相似的缓存
- **Prompt 模板**：预编译常用 Prompt

**工具选型**：
- **Redis**：最常用，功能丰富
- **Memcached**：简单场景
- **本地缓存**：`functools.lru_cache`

### 3.6 日志与监控

**要记录什么？**
- **每次调用**：时间、token 数、费用、延迟
- **错误信息**：错误码、堆栈、上下文
- **业务指标**：成功率、用户量、异常率

**工具推荐**：
- **结构化日志**：loguru、structlog
- **APM 监控**：Sentry、DataDog
- **可视化面板**：Grafana

### 3.7 异步与队列

处理大批量任务或用户量大时必备。

**典型场景**：
- 批量处理 1000 个文档
- 用户量大，AI 调用慢
- 需要错峰处理

**方案选择**：
- **Celery**：Python 最常用
- **Redis Queue**：轻量
- **云服务队列**：AWS SQS、阿里云 MNS

### 3.8 流式输出（Streaming）

AI 一边生成一边返回内容，用户体验大幅提升。

**实现示例**：

```python
response = openai.ChatCompletion.create(
    model="gpt-3.5-turbo",
    messages=[...],
    stream=True  # 关键参数
)
for chunk in response:
    print(chunk.choices[0].delta.content or "", end="")
```

**前端配合方式**：
- **SSE（Server-Sent Events）**：最常用
- **WebSocket**：双向通信
- **HTTP/2**：现代协议

### 3.9 上下文管理策略

超出窗口就要"忘记"前面的内容，需要智能管理。

**常见策略**：
- **滑动窗口**：只保留最近 N 轮对话
- **摘要压缩**：把长对话压缩成摘要
- **关键信息提取**：只保留关键内容
- **分层存储**：近期用完整，远期用摘要

## 四、成本与计费

### 4.1 AI API 的定价核心

> ⚠️ **最常见的认知误区**：很多新手以为买了"128K 套餐"就能每次用 128K，**其实不是！**
>
> 实际计费方式是按**实际使用的 token 数**收费，窗口大小只是"上限"，不是"配额"。

### 4.2 成本组成

| 成本项 | 说明 |
|--------|------|
| **输入 Token 费用** | 你发给 AI 的内容 |
| **输出 Token 费用** | AI 回复的内容（通常贵 2-3 倍） |
| **Embedding 费用** | RAG 用，文本转向量 |
| **微调费用** | 训练专属模型 |
| **向量数据库费用** | 云服务按存储和查询收费 |

### 4.3 开发者真实账单示例

假设每天 1000 个用户的 AI 应用：

| 使用场景 | 每次消耗 | 每天总消耗 | GPT-3.5 月费 | GPT-4 月费 |
|---------|---------|-----------|------------|----------|
| 简单聊天 | 500 token | 50 万 | 1 元 | 30 元 |
| 写邮件/文案 | 2000 token | 200 万 | 4 元 | 120 元 |
| 文档分析 | 10000 token | 1000 万 | 20 元 | 600 元 |
| 长文档处理 | 50000 token | 5000 万 | 100 元 | 3000 元 |

**关键洞察**：同样是 1000 用户/天，不同场景成本差 40 倍，这就是为什么很多 AI 应用"看起来便宜，实际烧钱"。

### 4.4 模型选型指南

| 模型 | 价格 | 适合场景 |
|------|------|---------|
| GPT-3.5 | 便宜 | 聊天、简单问答、写代码片段 |
| GPT-4 | 贵（128K 是 8K 的 2-3 倍） | 复杂推理、长文档 |
| Claude | 中高端 | 超长文档、整本书分析 |
| 国内模型（Qwen、DeepSeek） | 性价比高 | 中文场景、合规要求 |

结合 1.6 节的模型参数和 1.7 节的 Ollama，选型时还需要考虑：
- **参数规模**：1B~3B 适合本地实验，7B~14B 适合个人工具，70B+ 适合生产级应用
- **本地 vs 云端**：数据敏感场景用 Ollama 本地跑，追求速度和效果用云端 API

### 4.5 成本优化策略

**实操优化清单**：
- **精简 Prompt**：去掉废话，能说清楚的别啰嗦
- **按需选模型**：能 3.5 解决的别用 4
- **缓存结果**：相同问题不重复调用
- **限制对话轮数**：对话别无限增长
- **流式输出**：用户能看到进度，体感更快
- **分级使用**：简单的用便宜模型，复杂的才用贵的
- **断对话重来**：聊太长的对话就结束重来

## 五、质量与安全

### 5.1 评估体系（Evaluation）

AI 输出不稳定，**必须量化评估**。

**评估方法**：

| 方法 | 优点 | 缺点 |
|------|------|------|
| **人工评估** | 最准确 | 贵、慢 |
| **LLM 评估**：用 AI 评 AI | 快 | 可能受模型偏见影响 |
| **自动化指标** | 快、可重复 | 指标不一定贴合业务 |
| **A/B 测试** | 真实场景对比 | 需要流量 |
| **回归测试** | 防止性能下降 | 需要维护测试集 |

**评估工具**：
- **LangSmith**：LangChain 官方
- **Phoenix**：Arize AI 开源
- **PromptTools**：微软开源

### 5.2 Prompt 版本管理

Prompt 改一个字，结果可能天差地别，**必须做好版本管理**。

**要管理的内容**：
- **Prompt 模板版本**（Git 管理）
- **测试用例集**（各种边界情况）
- **评估结果**（哪个版本好）
- **回滚机制**（出问题了能回退）

**工具**：
- **Git**：基础版本控制
- **Dify**：可视化 Prompt 管理
- **LangSmith**：追踪每次调用的 Prompt

### 5.3 安全防护

**常见风险**：

| 风险 | 说明 |
|------|------|
| **提示词注入** | 用户诱导 AI 泄露系统提示 |
| **越狱** | 绕过 AI 的安全限制 |
| **数据泄露** | 上传敏感数据给 AI |
| **滥用** | 刷接口、薅羊毛 |

**防护手段**：
- **输入过滤**：检测恶意输入
- **输出过滤**：检测敏感输出
- **限流防刷**：单用户单日调用上限
- **数据脱敏**：敏感信息替换为占位符
- **审计日志**：记录所有敏感操作

## 七、常见问题

### 问题一：买了"128K 套餐"为什么实际用不到？

**现象**：充值或购买了大窗口套餐，但开发时发现大部分请求只有几百到几千 token。

**原因**：
- AI API 按**实际使用 token 数**计费，不是按"套餐"计费
- 窗口大小是"上限"，不是"配额"
- 你的实际需求可能根本不需要 128K 那么大

**处理方式**：
- 重新评估真实业务场景
- 简单任务用小窗口模型（如 4K、8K）
- 复杂任务再用大窗口（如 32K、128K）
- 按需选择，不必为"以防万一"买最贵的

### 问题二：GPT-3.5 和 GPT-4 成本差多少？

**现象**：同样一个 AI 应用，用 GPT-4 比 GPT-3.5 贵很多。

**原因**：
- GPT-4 基础定价是 GPT-3.5 的 10-30 倍
- GPT-4-128K 版本比 8K 版本还贵 2-3 倍
- 输出 Token 比 Input Token 贵

**处理方式**：
- **分级策略**：简单任务用 3.5，复杂任务用 4
- **任务分诊**：先用小模型判断任务难度，再决定用哪个模型
- **批量处理**：相似任务合并请求
- **缓存常用回答**：避免重复调用

### 问题三：为什么 AI 会"一本正经地胡说八道"？

**现象**：AI 给出了语法通顺、看起来合理，但实际上是编造的内容。

**原因**：LLM 本质是"概率计算器"，不是"知识库"，它不知道"事实"，只懂"什么词该接什么词"。

**处理方式**：
- **引入 RAG**：让 AI 基于真实资料回答
- **要求标注来源**：让 AI 注明信息来源
- **承认不知道**：在 Prompt 中要求"不知道就说不知道"
- **人工审核**：关键业务场景加入人工复核

### 问题四：长对话时 AI "忘记"了前面说过什么？

**现象**：聊了几十轮后，AI 不记得之前讨论的内容。

**原因**：超出上下文窗口的内容会被"截断"，AI 看不到就"忘"了。

**处理方式**：
- **滑动窗口**：只保留最近 N 轮对话
- **摘要压缩**：把长对话压缩成摘要
- **关键信息提取**：只保留关键内容
- **分层存储**：近期用完整，远期用摘要
- **升级模型**：用上下文窗口更大的模型（如 128K）

```
       ↓

           └── 匹配算法（找相似的片段）
       ↓
第四层：Ollama — 本地运行模型的工具
       ↓
应用层：Copilot、Cursor、Kimi、ChatGPT……（具体产品）
```

## 九、关键理解汇总

1. **LLM 本质是概率计算，不是图书馆查资料**
2. **LLM 只能输出文字，不能做事；Agent 在 LLM 外面包了一层执行能力**
3. **RAG 不是技术，是流程的名字**
4. **切片和向量化是离线做的，不是在提问时才做**
5. **Embedding 模型是独立的，和 LLM 可以来自不同公司**
6. **Agent 本质就是大量 if-else，分支多、执行快 → 看起来像真人**
7. **Ollama 就是让你本地跑模型的工具，相当于在家做饭 vs 去餐厅吃饭**
8. **Token 的实物类比**：1K ≈ 1 页 A4 纸，100K ≈ 一本小说
9. **OpenAI 协议的本质**：基于 HTTP 的 RESTful API 设计标准，不是神秘技术
10. **大窗口的计费真相**：不是"套餐配额"，而是"使用上限"，按实际消耗计费
11. **AI 开发的工程化**：从"能跑"到"能上线"需要完整的工程能力（成本、稳定性、安全性、可维护性）

**后续延伸方向**：
- 深入学习 RAG 项目实战（向量数据库选型、检索优化）
- 探索 Agent 开发（LangChain、AutoGen 框架）
- 关注多模态能力（GPT-4V、Whisper、DALL·E）
- 研究 Fine-tuning 微调技术
- 跟踪前沿协议（MCP 等）

**关键认知**：AI 应用的开发门槛在降低，但**工程化能力的门槛在升高**。性能、成本、稳定性、安全性、可维护性，缺一不可。
